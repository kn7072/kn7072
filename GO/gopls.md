# Полный справочник команд `gopls` (executeCommandProvider)

Этот документ содержит описание каждой команды из списка `executeCommandProvider` языкового сервера `gopls`. Команды сгруппированы по логическим категориям для удобства.

🔗 **Официальная документация gopls (Commands):**  
[https://github.com/golang/tools/blob/master/gopls/doc/commands.md](https://github.com/golang/tools/blob/master/gopls/doc/commands.md)  
🔗 **Настройки и возможности gopls:**  
[https://github.com/golang/tools/blob/master/gopls/doc/settings.md](https://github.com/golang/tools/blob/master/gopls/doc/settings.md)

## 📦 Управление модулями и зависимостями (Module & Dependency Management)

| Команда                     | Описание                                                                                                           |
| :-------------------------- | :----------------------------------------------------------------------------------------------------------------- |
| `gopls.add_dependency`      | Добавляет новую зависимость в файл `go.mod`.                                                                       |
| `gopls.remove_dependency`   | Удаляет указанную зависимость из `go.mod`.                                                                         |
| `gopls.upgrade_dependency`  | Обновляет конкретную зависимость в `go.mod` до последней доступной версии.                                         |
| `gopls.check_upgrades`      | Проверяет наличие доступных обновлений для зависимостей в `go.mod`.                                                |
| `gopls.go_get_package`      | Выполняет `go get <package>` для добавления или обновления конкретного пакета.                                     |
| `gopls.tidy`                | Выполняет `go mod tidy`: удаляет неиспользуемые зависимости и добавляет недостающие в `go.mod` и `go.sum`.         |
| `gopls.update_go_sum`       | Принудительно обновляет контрольные суммы в файле `go.sum`.                                                        |
| `gopls.vendor`              | Выполняет `go mod vendor`, копируя все зависимости в локальную директорию `vendor/`.                               |
| `gopls.modules`             | Возвращает список всех Go-модулей в текущем рабочем пространстве (workspace).                                      |
| `gopls.edit_go_directive`   | Позволяет изменить директивы `go` или `toolchain` в файле `go.mod` (например, с `go 1.20` на `go 1.21`).           |
| `gopls.run_go_work_command` | Выполняет команды, связанные с Go Workspaces (`go.work`), например, добавление или удаление модулей из воркспейса. |

## 🛠 Рефакторинг и редактирование кода (Refactoring & Editing)

| Команда                     | Описание                                                                                                           |
| :-------------------------- | :----------------------------------------------------------------------------------------------------------------- |
| `gopls.add_import`          | Добавляет недостающий оператор `import` в текущий файл (часто вызывается автоматически через Code Actions).        |
| `gopls.scan_imports`        | Сканирует файл и исправляет/организует импорты (аналог "Organize Imports").                                        |
| `gopls.list_imports`        | Возвращает список всех импортов в текущем файле (используется внутренними механизмами или плагинами).              |
| `gopls.list_known_packages` | Возвращает список всех пакетов, известных `gopls` в текущем воркспейсе (используется для автодополнения импортов). |
| `gopls.change_signature`    | Рефакторинг: изменяет сигнатуру функции (имена/типы параметров) и автоматически обновляет все места её вызова.     |
| `gopls.extract_to_new_file` | Рефакторинг: извлекает выделенный блок кода в новую функцию или новый файл.                                        |
| `gopls.apply_fix`           | Применяет автоматическое исправление к коду (например, запуск аналога `go fix` для устаревшего API).               |

## 🧪 Тестирование и генерация кода (Testing & Generation)

| Команда                | Описание                                                                                                                  |
| :--------------------- | :------------------------------------------------------------------------------------------------------------------------ |
| `gopls.add_test`       | Генерирует шаблон (stub) тестовой функции для функции или метода, находящегося под курсором.                              |
| `gopls.run_tests`      | Запускает Go-тесты для текущего файла или пакета. Может принимать аргументы для запуска конкретного теста.                |
| `gopls.generate`       | Выполняет команду `go generate` для текущего пакета или файла (полезно для генерации моков, кода из прото-файлов и т.д.). |
| `gopls.regenerate_cgo` | Принудительно перегенерирует файлы и заглушки для `cgo` (полезно, если изменились заголовочные файлы C).                  |

| Команда                          | Описание                                                                                                             |
| :------------------------------- | :------------------------------------------------------------------------------------------------------------------- |

| :----------------------------- | :-------------------------------------------------------------------------------------------------------------------- |
| `gopls.run_govulncheck`        | Запускает инструмент `govulncheck` для текущего воркспейса, чтобы найти известные уязвимости в зависимостях или коде. |

## ⚙️ Внутренние, отладочные и профильные команды (Internal, Debugging & Profiling)

_⚠️ Эти команды в основном предназначены для разработчиков самого `gopls` или для глубокой отладки. Обычному пользователю они редко нужны._

| Команда                            | Описание                                                                                                                     |
| :--------------------------------- | :--------------------------------------------------------------------------------------------------------------------------- |
| `gopls.assembly`                   | Показывает сгенерированный ассемблерный код для функции под курсором (использует `go tool compile -S`).                      |
| `gopls.gc_details`                 | Показывает детали работы сборщика мусора (Garbage Collection) и подсказки по оптимизации выделения памяти для текущего кода. |
| `gopls.mem_stats`                  | Возвращает статистику использования памяти самим процессом `gopls`.                                                          |
| `gopls.workspace_stats`            | Возвращает статистику воркспейса: количество проанализированных файлов, пакетов и т.д.                                       |
| `gopls.views`                      | Возвращает список активных "представлений" (views) или конфигураций, которыми управляет `gopls` в текущем воркспейсе.        |
| `gopls.start_profile`              | Запускает профилирование (CPU или Memory) самого сервера `gopls` для поиска узких мест в его производительности.             |
| `gopls.stop_profile`               | Останавливает активное профилирование сервера `gopls` и сохраняет результаты.                                                |
| `gopls.start_debugging`            | Внутренняя команда для инициации сессии отладки (обычно используется плагинами DAP, такими как `nvim-dap`, а не вручную).    |
| `gopls.client_open_url`            | Просит клиент (Neovim/VSCode) открыть указанный URL в браузере (например, ссылку на документацию).                           |
| `gopls.add_telemetry_counters`     | (Внутренняя) Добавляет счетчики для системы сбора телеметрии `gopls`.                                                        |
| `gopls.maybe_prompt_for_telemetry` | (Внутренняя) Проверяет, нужно ли показать пользователю запрос на включение телеметрии.                                       |

## 💡 Как узнать аргументы для конкретной команды?

Поскольку `workspace/executeCommand` требует специфичных аргументов, их нельзя угадать. Чтобы узнать, что передать в поле `arguments`:

1. Откройте официальную документацию: [gopls commands.md](https://github.com/golang/tools/blob/master/gopls/doc/commands.md)
2. Найдите нужную команду (например, `gopls.run_tests`).
3. В документации будет указан тип аргумента. Например, для `gopls.run_tests` аргументом часто является объект с полями `URI` (файла или директории) и `Debug` (boolean).

**Пример правильного вызова с аргументами (для `gopls.run_tests`):**

```lua
client.request('workspace/executeCommand', {
    command = 'gopls.run_tests',
    arguments = { { URI = vim.uri_from_bufnr(0), Debug = false } }
}, function(err, result)
    -- обработка результата
end)

# Справочник команд `gopls` и их терминальные аналоги

Отличный запрос! Это самый правильный способ понять, что на самом деле делают эти команды.

Многие команды `gopls` — это просто удобные обертки над стандартными утилитами Go (`go mod`, `go test`, `go doc`). Другие же являются сугубо внутренними функциями IDE и не имеют прямого аналога в терминале.

Ниже приведен полный разбор: для каждой команды указан **точный эквивалент для выполнения в обычном терминале (bash/zsh)**.

---

## 📦 1. Управление модулями и зависимостями (Прямые аналоги CLI)

Эти команды `gopls` просто вызывают стандартные команды `go` под капотом.

| Команда `gopls` | Эквивалент в терминале (Bash/Zsh) | Что делает |
| :--- | :--- | :--- |
| `gopls.tidy` | `go mod tidy` | Удаляет неиспользуемые и добавляет недостающие зависимости. |
| `gopls.add_dependency` | `go get <путь_к_пакету>` | Добавляет пакет в `go.mod` и скачивает его. |
| `gopls.remove_dependency` | `go mod edit -droprequire=<пакет>` <br> `go mod tidy` | Удаляет требование к пакету из `go.mod` и очищает кэш. |
| `gopls.upgrade_dependency` | `go get <пакет>@latest` | Обновляет конкретный пакет до последней версии. |
| `gopls.check_upgrades` | `go list -u -m all` | Показывает список всех модулей и доступные для них обновления. |
| `gopls.go_get_package` | `go get <пакет>` | То же самое, что и `add_dependency`. |
| `gopls.vendor` | `go mod vendor` | Копирует все зависимости в папку `vendor/` в корне проекта. |
| `gopls.update_go_sum` | `go mod download` <br> `go mod verify` | Скачивает модули и проверяет контрольные суммы в `go.sum`. |
| `gopls.edit_go_directive` | `go mod edit -go=1.22` | Меняет минимальную требуемую версию Go в `go.mod`. |
| `gopls.run_go_work_command`| `go work use ./my_module` | Добавляет модуль в файл `go.work` (для Go Workspaces). |
| `gopls.modules` | `go list -m all` | Выводит список всех модулей в графе зависимостей. |

---

## 🛠 2. Работа с кодом и рефакторинг

Для этих задач в терминале используются отдельные утилиты из инструментария Go или сторонние CLI-инструменты.

| Команда `gopls` | Эквивалент в терминале (Bash/Zsh) | Что делает |
| :--- | :--- | :--- |
| `gopls.add_import` <br> `gopls.scan_imports` | `goimports -w <файл.go>` | Автоматически добавляет недостающие импорты и форматирует их. *(Требует установки: `go install golang.org/x/tools/cmd/goimports@latest`)* |
| `gopls.apply_fix` | `go fix ./...` | Обновляет устаревший код API до новых версий Go (аналог `gofix`). |
| `gopls.doc` | `go doc <пакет>.<Символ>` <br> *или* <br> `go doc -all <пакет>` | Выводит документацию (godoc) для функции, типа или пакета прямо в терминал. |
| `gopls.assembly` | `go build -gcflags="-S" <файл.go>` | Выводит сгенерированный ассемблерный код для указанного файла. |
| `gopls.generate` | `go generate ./...` | Запускает все директивы `//go:generate` в проекте (для моков, protobuf и т.д.). |

---

## 🧪 3. Тестирование и безопасность

| Команда `gopls` | Эквивалент в терминале (Bash/Zsh) | Что делает |
| :--- | :--- | :--- |
| `gopls.run_tests` | `go test -v ./...` <br> *или для одного файла:* <br> `go test -v <файл_test.go>` | Запускает тесты с подробным выводом. |
| `gopls.run_govulncheck` <br> `gopls.vulncheck` | `govulncheck ./...` | Сканирует код и зависимости на известные уязвимости. *(Требует установки: `go install golang.org/x/vuln/cmd/govulncheck@latest`)* |

## ⚠️ 4. Команды, НЕ ИМЕЮЩИЕ прямого аналога в терминале

Эти команды существуют **исключительно** для взаимодействия IDE (Neovim, VS Code) с внутренним состоянием сервера `gopls`. В терминале им нет места, но я указал **ближайший возможный аналог** для отладки.

| Команда `gopls` | Почему нет аналога в терминале? | Ближайший аналог для терминала (если нужно) |
| :--- | :--- | :--- |
| `gopls.add_test` | Генерирует шаблон теста на основе AST текущего файла. | Установить `gotests`: `gotests -all -w <файл.go>` |
| `gopls.change_signature` | Рефакторинг с обновлением всех вызовов в проекте. | Установить `gorename`: `gorename -from '"pkg".Func' -to NewName` |
| `gopls.extract_to_new_file` | Выделение кода в новую функцию/файл через AST. | Только вручную или через специализированные CLI-рефакторы. |
| `gopls.diagnose_files` | `gopls` проверяет файлы в фоновом режиме. | `go build ./...` или `golangci-lint run` |
| `gopls.reset_go_mod_diagnostics` | Очистка внутреннего кэша ошибок `gopls`. | `go clean -cache` (очищает кэш сборки Go) |
| `gopls.regenerate_cgo` | Пересборка CGO-заглушек внутри `gopls`. | `go clean -cache && go build` |
| `gopls.start_profile` <br> `gopls.stop_profile` | Профилирование *самого процесса gopls*. | Запуск `gopls` вручную с флагом `-rpc.trace` или использование `pprof`. |
| `gopls.mem_stats` <br> `gopls.gc_details` <br> `gopls.workspace_stats` | Внутренняя статистика работы демона `gopls`. | `ps -o pid,rss,command -p $(pgrep gopls)` (посмотреть потребление RAM процессом). |
| `gopls.views` <br> `gopls.packages` <br> `gopls.package_symbols` | Внутреннее представление графа пакетов в памяти `gopls`. | `go list -json ./...` (выводит информацию о пакетах в JSON). |
| `gopls.list_imports` <br> `gopls.list_known_packages` | Кэшированный список импортов внутри `gopls`. | `grep -r "^import" .` или `go list -f '{{.Imports}}' ./...` |
| `gopls.free_symbols` | Поиск неиспользуемых символов в памяти `gopls`. | `staticcheck ./...` или `unused ./...` |
| `gopls.client_open_url` | Просит Neovim открыть браузер. | `xdg-open <url>` (Linux) или `open <url>` (macOS). |
| `gopls.start_debugging` | Инициализация DAP-сессии внутри IDE. | Запуск `dlv debug` (Delve) вручную в терминале. |
| `gopls.add_telemetry_counters` <br> `gopls.maybe_prompt_for_telemetry` | Внутренняя телеметрия Google. | Не применимо. |
| `gopls.fetch_vulncheck_result` | Чтение кэша уязвимостей `gopls`. | Просто запустить `govulncheck ./...` заново. |

## 💡 Главный вывод

Если вы видите команду `gopls.*`, которая управляет **проектом** (модули, тесты, зависимости), **всегда предпочитайте терминальный аналог** (`go mod tidy`, `go test`).

Используйте команды `gopls` через LSP (`client.request` или `code_action`) **только** тогда, когда действие требует понимания контекста кода (например, "добавить импорт *именно для этой переменной*" или "переименовать переменную *во всех файлах проекта*"). С этим справится только языковой сервер, работающий с AST, а не простой shell-скрипт.
```
