Общий контекст

Эти правила созданы инструментом sshuttle — он реализует «poor man’s VPN» через SSH‑туннель. Правила перенаправляют трафик через локальный прокси‑сервер (на порты 12 299 и 12 300), который затем отправляет его через SSH‑соединение.

Таблица: nat (Network Address Translation) — используется для изменения IP‑адресов и портов пакетов.
Цепочка: sshuttle-12300 — пользовательская цепочка, созданная sshuttle.
Действие -A: добавляет правило в конец указанной цепочки.
Разбор правил

1. UDP‑DNS‑запросы на локальный DNS (127.0.0.53)
   bash

-A sshuttle-12300 -d 127.0.0.53/32 -p udp -m udp --dport 53 -j REDIRECT --to-ports 12299

    -d 127.0.0.53/32 — пакет адресован на IP 127.0.0.53 (локальный DNS‑сервер, например, systemd-resolved).

    -p udp — протокол UDP.

    -m udp — загружает модуль сопоставления для UDP (позволяет использовать --dport).

    --dport 53 — порт назначения 53 (стандартный порт DNS).

    -j REDIRECT — действие: перенаправить пакет на другой порт на этом же хосте.

    --to-ports 12299 — перенаправить на порт 12 299 (вероятно, там слушает DNS‑прокси sshuttle).

Назначение: перехватить DNS‑запросы, отправленные на локальный резолвер 127.0.0.53, и перенаправить их на внутренний DNS‑прокси sshuttle (порт 12 299).

2. UDP‑DNS‑запросы на основной шлюз (192.168.1.1)
   bash

-A sshuttle-12300 -d 192.168.1.1/32 -p udp -m udp --dport 53 -j REDIRECT --to-ports 12299

    -d 192.168.1.1/32 — пакет адресован на IP роутера/шлюза (192.168.1.1).

    Остальные условия и действия идентичны правилу 1.

Назначение: перехватить DNS‑запросы, которые приложение пытается отправить напрямую на DNS‑сервер провайдера (часто прописан как IP шлюза), и тоже перенаправить их на прокси sshuttle (12 299). Это гарантирует, что все DNS‑запросы пойдут через туннель.

3. Возврат трафика для локальных адресов
   bash

-A sshuttle-12300 -m addrtype --dst-type LOCAL -j RETURN

    -m addrtype — загружает модуль для проверки типа адреса.

    --dst-type LOCAL — условие: IP‑адрес назначения является локальным для этого хоста (т. е. совпадает с одним из IP‑адресов сетевой карты).

    -j RETURN — действие: прекратить обработку в текущей цепочке и вернуться в вызывающую цепочку (в данном случае — в PREROUTING или OUTPUT).

Назначение: не трогать трафик, предназначенный для самого хоста. Это важно для корректной работы служб, слушающих на локальных адресах.

4. Возврат TCP‑трафика на 127.0.0.1
   bash

-A sshuttle-12300 -d 127.0.0.1/32 -p tcp -j RETURN

    -d 127.0.0.1/32 — пакет адресован на 127.0.0.1 (стандартный loopback‑адрес).

    -p tcp — протокол TCP.

    -j RETURN — прекратить обработку в цепочке.

Назначение: исключить из перенаправления TCP‑соединения, идущие на localhost (127.0.0.1). Это нужно, чтобы локальные приложения (веб‑серверы, базы данных и т. д.) продолжали работать без вмешательства sshuttle.

5. Возврат TCP‑трафика в локальную сеть
   bash

-A sshuttle-12300 -d 192.168.1.0/24 -p tcp -j RETURN

    -d 192.168.1.0/24 — пакет адресован в подсеть 192.168.1.0/24 (типичная домашняя/офисная локальная сеть).

    -p tcp — протокол TCP.

    -j RETURN — прекратить обработку.

Назначение: не перенаправлять трафик, идущий на устройства в локальной сети (принтеры, NAS, другие ПК). Он должен идти напрямую, а не через SSH‑туннель — это снижает нагрузку и задержки.

6. Перенаправление всего остального TCP‑трафика
   bash

-A sshuttle-12300 -p tcp -j REDIRECT --to-ports 12300

    -p tcp — условие: пакет использует протокол TCP (без других ограничений).

    -j REDIRECT — действие: перенаправить.

    --to-ports 12300 — перенаправить на порт 12 300.

Назначение: последнее правило-ловушка. Оно перехватывает весь оставшийся TCP‑трафик, который не попал под предыдущие исключения (не локальный, не в LAN, не DNS). Этот трафик перенаправляется на порт 12 300 — вероятно, там работает основной TCP‑прокси sshuttle, который инкапсулирует его в SSH‑туннель.
Логика работы цепочки в целом

Порядок правил критичен. Пакеты обрабатываются сверху вниз. Цепочка реализует следующую логику:

    Перехватить DNS‑запросы (UDP/53), идущие на локальный резолвер или шлюз, и отправить их на DNS‑прокси (12 299).

    Пропустить локальный трафик (LOCAL, 127.0.0.1, локальная сеть 192.168.1.0/24) — он не должен идти через туннель.

    Перенаправить всё остальное (остальной TCP) на основной прокси‑порт (12 300) для отправки через SSH.

Это обеспечивает:

    Конфиденциальность DNS: все запросы идут через туннель, провайдер их не видит.

    Производительность: локальный и LAN‑трафик не нагружает туннель.

    Функциональность: приложения, работающие на localhost, продолжают работать штатно.
